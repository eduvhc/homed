Um homelab declarativo num único Beelink — docker-compose para o que corre em casa, OpenTofu para o que vive na Cloudflare, sops + age para tudo o que é segredo. Reproduzível noutra máquina em menos de uma hora.

E o resto?

O fluxo acima é o caminho principal. Estes três temas correm ao lado, independentes.

tema 01

DNS de casa

Em casa, quando ligo ao Wi-Fi, o telefone pergunta primeiro ao h-adguard. É um Pi-hole alternativo: bloqueia ads e resolve *.iedora.com para o IP local (em vez de ir à Cloudflare → fica muito mais rápido).

tema 02

Backup automático

Todas as noites o h-restic-backup tira um snapshot encriptado (configs, secrets, biblioteca de música) e envia para o R2 da Cloudflare. Ao Domingo verifica integridade e aplica retention.

tema 03

Está tudo vivo?

O h-gatus faz duas coisas: vai bater à porta de cada serviço (probe) e espera receber um "heartbeat" das jobs cron (deadman). Se algo falhar, envia push via ntfy.sh para o telemóvel.

tema 04

Como tudo é provisionado (sem clicar em UIs)

Tudo o que vês acima nasce de ficheiros num repo Git. Quatro ferramentas tratam disto. Migrar para outra máquina = clonar o repo + correr task up.

tema 05

Como uma app SaaS arranca (sem ninguém mexer numa UI)

Cada app tem o seu par de sidecars: <app>-db-bootstrap cria o user e a DB no h-postgres, <app>-db-migrate aplica os SQLs (dbmate). Só depois é que a app arranca. Tudo declarativo, idempotente, num único docker compose up -d.

Porquê estas escolhas?

Cada decisão tem alternativas legítimas. Aqui está o que rejeitei e porquê — para futuros eus e juniors que herdarem isto.

01
runtime

docker-compose, não NixOS + k3s

✓ docker-compose v2 + Debian 12
✗ NixOS + k3s + ArgoCD

Considerei seriamente NixOS+k3s — mais "puro", mais GitOps. Rejeitei porque amarra-me ao Nix: testar um dashboard novo em Nix é um pincel, e o critério decisivo era "reproduzível noutra máquina sem dor". Docker-compose corre em qualquer Linux (Synology, RPi5, Fedora) sem reescrever nada. Para single-node a complexidade de k8s não compensa.

02
reverse proxy

Caddy, não Traefik nem Nginx

✓ caddy-docker-proxy (labels)
✗ Traefik · Nginx

Caddy: config implícita por labels Docker, HTTPS automático, binário mais simples que Traefik. Traefik tem middleware ecosystem maior mas a complexity tax não vale a pena para 10 serviços. Nginx obriga a regenerar nginx.conf a cada serviço novo — atrito desnecessário.

03
sso

Authelia, não Authentik nem Pocket ID

✓ Authelia + forward-auth
✗ Authentik · Pocket ID · Keycloak

Authelia faz forward-auth — funciona com QUALQUER app por trás de proxy, mesmo apps que nunca ouviram falar de OIDC (a maioria do *arr stack). Pocket ID e Authentik focam-se em OIDC nativo, que a maior parte das self-hosted apps não suporta. Authelia também ganha por config 100% YAML versionado.

04
dns sinkhole

AdGuard Home, não Pi-hole

✓ AdGuard · AdGuardHome.yaml
✗ Pi-hole (gravity.db SQLite)

AdGuard guarda tudo em YAML versionável no Git — verdadeiramente declarativo. Pi-hole guarda config em SQLite opaca (gravity.db) — perdes a propriedade "git é source-of-truth". Funcionalmente equivalente para o que preciso.

05
cache

Valkey, não Redis

✓ Valkey 9.1 (BSD)
✗ Redis (RSALv2/SSPL)

Em 2024 a Redis mudou a licença para algo não-OSS. A Linux Foundation criou o fork Valkey, mantido pela comunidade. Drop-in compatível com Redis (Authelia nem se apercebe). Escolher Valkey é a decisão alinhada com o ethos self-hosted/OSS.

06
network ingress

Cloudflare Tunnel, não port-forwarding

✓ Zero-Trust Tunnel (QUIC)
✗ port-forward 443 no router · WireGuard

Zero portas abertas no router, zero IP público exposto. A ligação é de dentro para fora (saída do Beelink → CF edge). Bónus: DDoS protection grátis, geoblock fácil, IP do meu home invisível. WireGuard exigiria sempre o cliente instalado — quebra "abrir o link do telemóvel da rua".

07
iac

OpenTofu 1.10, não Terraform

✓ OpenTofu (Linux Foundation)
✗ Terraform (BSL)

Mesmo HCL, mesma toolchain, drop-in replacement. OpenTofu tem ainda state encryption nativa (1.10+) — posso commitar o terraform.tfstate encriptado ao Git sem leak. Terraform não tem isto. E a licença BSL da Hashicorp é hostil para self-hosters.

08
secrets

sops + age, não Vault nem Bitwarden

✓ sops + age (file-based)
✗ Hashicorp Vault · Bitwarden Secrets Manager

Vault é overkill para single-node — exige cluster, unseal, KMS. sops cifra ficheiros in-place no Git com chave age local. Zero infra extra, zero serviço a manter. A passphrase vive no Keychain. Trade-off honesto: sem audit trail centralizado, mas para 1 utilizador é aceitável.

09
secrets org

Secrets centralizados, não per-service

✓ ~/homed/secrets/*.env.sops
✗ secret dentro de stacks/h-X/.env.sops

Single Responsibility: secrets não fazem parte da definição de um serviço — são injectados em runtime. Manter num só sítio dá: rotação fácil (um task rotate), auditoria simples (ls secrets/), backup óbvio. Custo: paths feios ../../../../secrets/ nos compose files. Vale a pena.

10
backup engine

Restic, não Borg nem Duplicati

✓ Restic (S3-compatible)
✗ Borg (SSH-only) · Duplicati (Mono UI)

Restic faz encryption client-side + dedup + suporte nativo S3. Funciona contra R2 sem proxy. Borg exige um servidor SSH — não corre contra object storage. Duplicati tem UI em .NET/Mono — instável e não-declarativo. Restic é CLI puro: cabe num container, configurável por env vars.

11
backup target

Cloudflare R2, não S3 nem B2

✓ Cloudflare R2
✗ AWS S3 · Backblaze B2

R2 é S3-compatível mas sem egress fees — restaurar 100 GB não custa €9 como na AWS. Já estou no ecossistema Cloudflare (tunnel, DNS) → mesmo dashboard, mesmo billing. B2 é viável mas paga egress; para um setup só com 1 token de provider, R2 simplifica.

12
monitoring

Gatus sozinho, não Uptime Kuma + Healthchecks

✓ Gatus (probes + push num só)
✗ Uptime Kuma + Healthchecks.io

Inicialmente tinha os dois — Uptime Kuma para probes activos (bater à porta dos serviços) e Healthchecks para deadman switch (esperar pings). Dois serviços para o mesmo problema. Gatus faz ambos num só container, config 100% YAML versionada. Menos infra, mesma cobertura.

13
alerts

ntfy.sh, não Discord nem email

✓ ntfy.sh (free, app móvel)
✗ Discord webhook · SMTP · Pushover

ntfy.sh é OSS, tem app móvel grátis (Android/iOS), permite topic privado, zero conta. Discord exige conta e mistura alerts com chat social. SMTP exige um servidor email — outro componente a manter. Pushover é pago.

14
provisioning

Ansible, não bash script

✓ ansible/provision.yml
✗ provision.sh

O primeiro provision.sh tinha 80 linhas de apt install + curl | sh. Não era idempotente: correr 2× partia coisas. Ansible é declarativo — descreve o estado final, decide o que fazer. Reaplica sem medo, mostra exactly o que mudou, e o playbook documenta o setup. "Scripts em sh hmmm" foi a tua reacção — and you were right.

15
task runner

go-task, não Make nem just

✓ Taskfile.yaml (go-task)
✗ Makefile · justfile

Make: sintaxe arcana, tabs sagrados, dependências por filesystem. just: melhor, mas DSL própria. go-task: YAML, igual ao resto do projecto (compose, ansible, gatus). Vars, includes e deps em YAML simples — um junior lê e percebe.

16
credentials

Uma só credencial fora do repo

✓ Keychain → CLOUDFLARE_API_TOKEN
✗ .env local · vault separado · multiplos tokens

Tudo o que é segredo já está cifrado em secrets/*.env.sops. A única coisa que não pode estar lá é a passphrase para decifrar. Solução: vive no Keychain do macOS, único ponto de "trust me". Migrar para nova máquina = importar 1 password + 1 chave age.

17
postgres

Postgres partilhado, 1 role por DB

✓ 1 instância h-postgres + 1 role por DB
✗ Postgres per-app · 1 superuser partilhado entre apps

1 instância serve todas as apps SaaS (~50 MB idle por instância — pôr 3 não muda nada num Beelink, mas multiplica upgrades, backups e operação). Per-app teria ganhos reais de isolação se houvesse equipa grande; para single-operator é over-engineering. O ponto fraco do partilhado — uma app patológica afecta as outras — mitigamos com 1 role por DB em vez de superuser único: se imopush é comprometida, não lê core. statement_timeout por role mitiga query patológica.

18
migrations

dbmate + h-bootstrap-db, não Drizzle nem init.d

✓ sidecars <app>-db-bootstrap + <app>-db-migrate (dbmate)
✗ Drizzle como motor canónico · init.d hardcoded · docker exec à mão

Drizzle pode continuar como ORM/query-builder de runtime — não é problema do homed. Para migrations versionadas a ferramenta canónica é dbmate (SQL puro, 1 ficheiro por migration, idempotente). Bootstrap (CREATE ROLE+DATABASE) é o passo que o dbmate não faz: pre-requisito num container caseiro mínimo (15 linhas de shell, idempotente). Pattern: bootstrap → migrate → app ligado por service_completed_successfully. Resultado: docker compose up sobe tudo do zero numa máquina nova, sem intervenção humana.

Princípios

  • 01 · reproduzível noutra máquina em < 1h
  • 02 · git é o source-of-truth — UI state proibido
  • 03 · uma só credencial fora do repo (keychain)
  • 04 · SRP — secrets centralizados, nunca por-serviço
  • 05 · pin de versões — zero :latest
  • 06 · nada exposto no host — tudo via Caddy + Authelia
